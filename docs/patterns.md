# Патерни проєктування, SOLID та бізнес-логіка

## Патерни GoF

| Патерн | Призначення в системі | Файли |
| --- | --- | --- |
| **Strategy** | Алгоритм оцінювання обирається за іменем; сервіс застосовує його, не знаючи конкретної реалізації. Додати нову схему = додати тип, без зміни викликачів. | `internal/grading/strategy.go`, `weighted.go`, `passfail.go`, `letter.go` |
| **Factory** | Єдине місце, що збирає базовий курс, навішує декоратори й валідує стратегію. Використовується і при створенні, і при відновленні з БД — гарантує узгодженість. | `internal/course/factory.go` |
| **Decorator** | Курс розширюється поведінкою (сертифікат, онлайн-платформа), лишаючись `domain.Course`. Декоратори стекуються детерміновано. | `internal/course/decorator.go` |
| **Observer** | Після збереження оцінки публікується подія; підписники (трекер прогресу, логер) реагують незалежно. Нові реакції додаються без зміни сервісу. | `internal/progress/observer.go`, `tracker.go`, `notifier.go` |
| **Repository** | Бізнес-логіка залежить від інтерфейсів сховища; SQLite-реалізації впроваджуються ззовні, тож бекенд змінний. | `internal/service/ports.go`, `internal/storage/*.go` |

## Відповідність SOLID

- **S (Single Responsibility)** — кожен пакет має одну причину змінюватись:
  `grading` рахує оцінки, `storage` зберігає, `httpapi` транслює HTTP, `service`
  оркеструє. `main` — єдине місце композиції.
- **O (Open/Closed)** — нова стратегія оцінювання чи новий спостерігач додаються
  без редагування наявного коду (реєстр стратегій + список спостерігачів).
- **L (Liskov)** — будь-який `Course` (базовий чи декорований) взаємозамінний;
  декоратори зберігають контракт інтерфейсу.
- **I (Interface Segregation)** — інтерфейси вузькі: `courseLookup` має лише
  `Get`, `progress.Store` — лише `SetProgress`. Споживач залежить тільки від
  потрібних методів.
- **D (Dependency Inversion)** — `service` оголошує інтерфейси та залежить від
  абстракцій; конкретику постачає `cmd/server`.

## Складна бізнес-логіка (алгоритми)

1. **Зважене середнє** (`WeightedAverage`): підсумовує `percent × weight`, ділить
   на суму ваг; за відсутності ваг коректно деградує до простого середнього,
   уникаючи ділення на нуль.
2. **Похибка округлення** (`round2`): фінальна оцінка округлюється до двох знаків
   для стабільного зберігання.
3. **Літерна шкала** (`letterFor`): відображення 0..100 → A–F з порогами.
4. **Обчислення прогресу** (`computeProgress`): частка оцінених робіт від
   запланованих із кліпуванням до 1; за невідомого плану — фолбек на pass/fail.
5. **Перехід станів зарахування**: `pending → active → completed` залежно від
   співвідношення `graded`/`planned` (див. state diagram в `architecture.md`).

Усі алгоритми покриті табличними тестами з перевіркою граничних значень
(порожні дані, нульовий максимум, межі літерних оцінок, перевищення плану).
